# v3.1 缓存跨会话丢失 —— 问题追踪手记

## 问题描述

BubbleTrans 打开了压缩包（.cbz/.zip），翻译了很多页，关闭程序后再打开同样的压缩包，之前的翻译缓存全部失效，F5 会重新请求 API。

单次会话内缓存正常（F5 命中、预翻译命中、批量翻译都正常）。跨会话必现。

## 已完成的修复（4 层，逐个叠加，仍不生效）

### 修复 1：压缩包翻译缓存从不写入（已解决）
- **文件**: `src/engine/translation_controller.py`
- **根因**: `_on_finished()` 中用 `tempfile.gettempdir() in current_path` 判断是否为临时文件，压缩包解压到 `C:\Users\xxx\AppData\Local\Temp\` 下被误判
- **修复**: 新增 `_is_temp_image` 标志位，`translate_page()` 设 `False`，`translate_region()` 设 `True`

### 修复 2：退出时 WAL 数据丢失（已解决）
- **文件**: `src/utils/database.py`, `src/utils/cache.py`, `src/ui/window.py`
- **根因**: SQLite WAL 模式，`close()` 时不做 checkpoint，WAL 文件数据未写入主 DB
- **修复**: `Database.close()` 加 `PRAGMA wal_checkpoint(TRUNCATE)`，`closeEvent` 中调用 `cache.close()`

### 修复 3：压缩包每次解压到随机临时目录（已解决）
- **文件**: `src/utils/archive.py`
- **根因**: `tempfile.mkdtemp(prefix="bubbletrans_arc_")` 每次生成随机后缀目录，如 `bubbletrans_arc_a1b2c3` vs `bubbletrans_arc_d4e5f6`
- **修复**: 新增 `_get_archive_temp_dir()`，用 `MD5(archive_path)[:12]` 生成固定目录名

### 修复 4：重新解压导致 mtime 子秒级漂移（已解决）
- **文件**: `src/utils/cache.py` (`_make_key` 方法)
- **根因**: ZIP 时间戳精度 2 秒，但 `os.path.getmtime()` 返回微秒级 float。重新解压后同一文件的 mtime 有 ~0.1 秒的子秒级差异，导致 `MD5(path + mtime)` key 不匹配
- **诊断数据**: 同一文件两次解压 mtime 分别为 `1783668579.8645203` vs `1783668579.9618385`
- **修复**: `_make_key()` 中 `mtime` 取整：`raw = f"{image_path}:{int(mtime)}"` 替代 `f"{image_path}:{mtime}"`

## 修复 4 后仍不生效 —— 待排查

修复 4 的独立诊断脚本（`_diag.py`）已验证 key 匹配、缓存命中。但实际程序运行中跨会话仍失败。

### 可能的未排查方向

1. **`shutil.rmtree` + 重新解压后的实际 mtime**: 实际程序执行 `rmtree(temp_dir)` → `os.makedirs` → `zf.extract()`。如果 `rmtree` 失败（目录被锁定），文件保留旧 mtime；如果成功删除再重解压，文件是新的。`int(mtime)` 应不受影响，但需实际验证日志。

2. **`archive_path` 跨会话差异**: `_get_archive_temp_dir(archive_path)` 的入参来自 `FileBrowser._load_source(path)`. 如果 `path` 在两会话间不同（如相对 vs 绝对路径、大小写不一致），会导致不同的 temp 目录。检查 `FileBrowser._source_path` 的设置。

3. **`TranslationCache.__init__` 中的旧 JSON 检测**: 第 67 行 `if os.path.exists(json_path):` 检测 `translation_cache.json` 是否存在。如果残留同名文件会触发 `migrate_from_json()` 用 `INSERT OR REPLACE` 覆盖数据。检查 `D:\web\BubbleTrans\translation_cache.json` 是否残留。

4. **`Database.__init__` 的 `_init_schema()` 是否有副作用**: `executescript("CREATE TABLE IF NOT EXISTS ...")` 内部会隐式 COMMIT。在 WAL 模式下，新连接打开时是否可能丢弃未 checkpoint 的 WAL 数据？需要验证 `check_same_thread=False` + WAL 的组合在新连接场景下的行为。

5. **缓存写入的实际调用方差异**: `_on_finished()`(controller) vs `_on_worker_finished()`(prefetch) vs `_write_cache()`(batch)。检查这三条路径写入时 `image_path` 是否完全一致、`mtime` 采集时机是否一致。

6. **`TranslationCache` 是否被多次实例化**: `window.py:299` 创建一次。但检查是否有其他模块也创建了 `TranslationCache()` 实例指向不同的 db_path。
