## 问题 1：压缩包翻译缓存永不写入

**现象**：打开 .cbz 压缩包 → F5 翻译 → 翻到其他页再翻回 → 按 F5 → API 重新请求（缓存未命中）

**根因**：`translation_controller._on_finished()` 第 85 行用 `tempfile.gettempdir() in current_path` 判断是否为临时文件。压缩包解压到 `C:\...\Temp` 目录下，被误判为不可缓存的临时文件，`cache.set()` 从未执行。

**修复**：新增 `_is_temp_image` 标志位替代路径推断。
- `translate_page()` → `_is_temp_image = False`（可缓存）
- `translate_region()` → `_is_temp_image = True`（不缓存）
- `_on_finished()` → 用 `if not self._is_temp_image:` 替代 `if not is_temp:`

**改动的文件**：`src/engine/translation_controller.py`

## 问题 2：批量翻译 UI 卡顿且无进度反馈

**现象**：
- 批量翻译范围选择 → 耗时确认后，看不到进度条，不知道后台在干嘛
- 整个软件卡顿，鼠标移动、滚轮缩放都延迟

**根因**：
1. `_show_batch_progress()` 在 `start()` 之后调用，而 `start()` 是同步阻塞的——进度条永远没机会显示
2. `_process_chunk()` 中用 `event.wait(timeout=0.5)` 轮询，主线程每 500ms 才泵一次事件，UI 以 2fps 刷新
3. 每次 `_write_cache()` 都写 INFO 日志，大量 I/O

**修复**：
1. `_show_batch_progress()` 移到 `start()` 之前调用
2. poll 间隔从 500ms 降到 20ms（50fps），每个 chunk 后追加 `processEvents()`
3. 批量模式下 `_write_cache` 静默，不输出 INFO 日志

**改动的文件**：`src/ui/batch.py`、`src/ui/window.py`

## 问题 3：缓存退出时丢失

**现象**：翻译若干页 → 关闭程序 → 重启 → 之前翻译的缓存全部归零

**根因**：SQLite WAL 模式下，写入数据暂存在 `-wal` 日志文件中，需执行 `PRAGMA wal_checkpoint(TRUNCATE)` 才能写入主 DB 文件。`Database.close()` 仅关闭连接，不做 checkpoint——WAL 文件在下次连接时被丢弃。

实际数据印证：测试中 WAL 文件 4.1MB，主 DB 文件仅 4KB。

**修复**：
1. `Database.close()` → 加 `PRAGMA wal_checkpoint(TRUNCATE)` 确保刷盘
2. `TranslationCache.close()` → 新增，封装 `_db.close()`
3. `window.closeEvent` → 退出时调用 `self.translation_cache.close()`

**改动的文件**：`src/utils/database.py`、`src/utils/cache.py`、`src/ui/window.py`

## 问题 4：压缩包缓存跨会话丢失

**现象**：压缩包翻译若干页 → 关闭程序 → 重启打开同一压缩包 → 点击翻译 → API 重新请求

**根因**：压缩包每次解压使用 `tempfile.mkdtemp()`（随机后缀），同一压缩包每次会话解压到不同的临时目录。缓存 key = `MD5(full_path + mtime)`，路径变化 → key 不匹配 → 缓存全部失效。

实际：WAL checkpoint 正常（数据持久化到了 DB），但新 key 查不到旧 key 的数据。

**修复**：
1. `archive.py` 新增 `_get_archive_temp_dir()`，用 `MD5(archive_path)` 生成固定的临时目录名。每次解压前清理旧目录再重新创建。
2. `cache.py._make_key()` 将 `mtime` 取整（`int(mtime)`），消除重新解压时的子秒级漂移。

**改动的文件**：`src/utils/archive.py`、`src/utils/cache.py`
