## 动机

v2.4 功能完整可用，但 `window.py` 已达 1323 行，承载 15 种职责（UI 布局、文件管理、图片缓存、翻译编排、预翻译、导航、缓存清除等），且 `prefetch.py` 反向依赖 `window.TranslationWorker`。继续堆功能将导致代码不可维护。

v3.0 是一次纯架构手术——不做任何用户可见的功能变更，仅拆解 God Object，建立清晰三层分层。

## 重构后架构

```
src/
├── engine/                        ← 业务引擎层（不依赖 UI）
│   ├── workers.py                 ← 后台线程（TranslationWorker, ImageLoadWorker）
│   ├── translation_controller.py  ← 翻译编排（Worker 生命周期、缓存写回）
│   ├── llm.py                     ← 纯翻译逻辑
│   └── base64_vision.py
├── ui/                            ← 纯 UI 层（不包含业务逻辑）
│   ├── window.py                  ← 主窗口（仅布局 + 信号连接）
│   ├── canvas.py                  ← 图片画布
│   ├── image_cache.py             ← 图片 LRU 缓存管理器
│   ├── file_browser.py            ← 文件浏览器组件
│   └── prefetch.py                ← 预翻译管理器
└── utils/                         ← 工具层
    ├── cache.py                   ← 翻译缓存（已加锁）
    ├── config.py                  ← 配置读写（已加缓存）
    └── archive.py                 ← 压缩包解压
```

## 经验教训

1. **串行执行必要**：R2/R3/R4 都修改 `window.py` 的重叠区域（`__init__`、`closeEvent`、回调），实际执行中确认必须串行逐步合并，避免冲突。
2. **信号替代直接调用**：FileBrowser 和 ImageCacheManager 都通过 Qt 信号与 window.py 通信，这是正确的解耦方式。
3. **小粒度计划有效**：每个子任务独立计划文档使 AI 辅助实施更精准，偏差可控。
4. **顺手修复不应延后**：R9-R11 虽标注"最后做"，但实际实施中对架构无影响，可在 R4 后立即完成。
