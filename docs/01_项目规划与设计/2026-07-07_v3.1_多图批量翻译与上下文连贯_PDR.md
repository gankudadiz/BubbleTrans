## 1. 升级动机

v3.0 完成了架构重构后，代码有了干净的分层。v3.1 是这一轮升级的"用户感知层"——在健康架构上实现三个长期被诟病的问题：

### 1.1 页面是孤岛

每页翻译完全独立，第 5 页和第 50 页不知道彼此存在。导致的后果：

- **角色名漂移**：冷门角色在不同页面可能被译成不同名字（不是蝙蝠侠这种大 IP，而是独立漫画里只出现几次的角色）
- **剧情断层**：每页 AI 从零开始理解故事，缺乏整体语感
- **术语不一致**：同一个地名/组织名/招式名在不同页面可能译法不同

### 1.2 批量操作缺失

v2.4 的预翻译（`PrefetchManager`）只能预先翻译后续 1/3/5 页。用户如果想翻整本漫画，需要逐页点过去。200 页的漫画就是 200 次手动操作。

### 1.3 存储层触碰天花板

当前翻译缓存是单文件 JSON（`translation_cache.json`），启动时全量加载到内存 `OrderedDict`，每次修改后全量序列化落盘。跨页查询（"取第 10-14 页的剧情总结"、"查某术语的历史译法"）在 JSON 上只能全表遍历。此外，多并发写入时存在覆盖风险。

## 3. 核心设计

### 3.1 存储层迁移：JSON → SQLite

**选型理由**：不再重复讨论。`sqlite3` 是 Python 标准库，零额外依赖。

**Schema 设计**（最小化原则——只建必要字段和索引）：

```sql
-- 翻译缓存表
CREATE TABLE translations (
    cache_key   TEXT PRIMARY KEY,       -- MD5(image_path + mtime)
    folder_path TEXT NOT NULL,          -- 所属文件夹，用于按漫画隔离查询
    page_index  INTEGER,               -- 在文件夹中的图片序号（0-based）
    image_path  TEXT NOT NULL,          -- 图片绝对路径
    original    TEXT,                   -- 原文（OCR 结果）
    translated  TEXT,                   -- 译文
    plot        TEXT,                   -- 剧情总结（原 summary.plot）
    notes       TEXT,                   -- 备注（原 summary.notes）
    updated_at  TEXT NOT NULL           -- ISO 时间戳
);
CREATE INDEX idx_folder_page ON translations(folder_path, page_index);
```

**迁移策略**：
1. 启动时检测 `translation_cache.json` 存在且 SQLite 中无对应数据 → 自动执行迁移
2. 迁移完成后将 JSON 重命名为 `.bak`，不出错不删除
3. 之后所有读写走 SQLite，JSON 文件不再使用

**LRU 淘汰**：当前缓存上限 500 条。SQLite 方案改为按 `updated_at` 排序，超出上限时删除最旧记录。上限可在配置中调整（默认 2000 条，单个漫画 200 页足够 + 多本漫画的余量）。

### 3.2 多图批量翻译

**流程设计**：

```
┌─────────────────────────────────────────────────────┐
│                 批量翻译流程                           │
├─────────────────────────────────────────────────────┤
│                                                       │
│  ① 用户选择范围                                       │
│     工具栏[批量翻译...] → 弹出对话框:                   │
│       - 范围: [第 5 页] ~ [第 25 页] (共 21 页)       │
│       - 选项: ☑ 启用上下文连贯                        │
│       - 提示: ">15 页时自动提醒耗时"                    │
│                                                       │
│  ② 用户确认后，开始分块执行                            │
│     总页数 ≤ 15: 不弹提示，直接开始                     │
│     总页数 > 15: 弹提示"将翻译 XX 页，预计约 X-XX 分钟" │
│                                                       │
│  ③ 分块调度 (chunk_size=5, 并发=2)                    │
│     Chunk 1 (5-9):   context = 空                     │
│     Chunk 2 (10-14): context = 第5-9页的plot          │
│     Chunk 3 (15-19): context = 第10-14页的plot        │
│     Chunk 4 (20-24): context = 第15-19页的plot        │
│     Chunk 5 (25):    context = 第20-24页的plot        │
│                                                       │
│  ④ 进度反馈                                           │
│     右侧面板底部 → 进度条 + 文字 "12 / 21 页已完成"     │
│     可暂停 / 可取消                                    │
│                                                       │
│  ⑤ 完成后汇总                                         │
│     "翻译完成: 21/21 成功, 0 失败                      │
│      已自动应用到当前页，可直接翻阅查看"                  │
│                                                       │
└─────────────────────────────────────────────────────┘
```

**关键约束**：

| 约束 | 值 | 说明 |
|------|-----|------|
| 单次最大页数 | 50 页（可配置） | 防止用户误选太多 |
| 分块大小 | 5 页 | 组内并发翻译 |
| 并发数 | 复用 Settings 中的预翻译并发数（默认 2） | 避免重新发明 |
| 耗时提示阈值 | 15 页 | 低于不提示，高于弹预估时间 |
| 上下文窗口 | 前 5 页（即上一个 chunk） | 5 页 summary 约 300-500 字 |
| 失败策略 | 标记失败 + 跳过，不影响后续 | 结束后列出失败页，支持重试 |

**UI 设计要点**：

- 入口：工具栏"预翻译 ▼"下拉菜单新增一项「批量翻译...」，或单独按钮
- 对话框：范围选择用两个 SpinBox（起始页 / 结束页），自动计算总页数
- 进度：右侧面板底部或独立浮动进度条，显示"已完成 X/Y，当前翻译第 Z 页"
- 可中断：暂停按钮保留已完成结果，取消按钮清空本次未写入的缓存

### 3.3 上下文传递

**核心机制**：翻译第 N 页时，将第 N-5 到 N-1 页的 `plot` 字段拼接为"前情提要"，注入到 LLM prompt 中。

> **v3.0 重构收益**：上下文引擎（`engine/context.py`）可以直接通过 SQLite 查询 `SELECT plot FROM translations WHERE folder_path=? AND page_index BETWEEN ? AND ? ORDER BY page_index`，不再需要遍历 JSON。

**Prompt 变化**（示意）：

```
现有 prompt:
  system: "你是专业美漫翻译..."
  user: [图片]

v3.1 prompt:
  system: "你是专业美漫翻译..."
  context: "【前情提要】
    第5页：布鲁斯与阿尔弗雷德在蝙蝠洞分析线索...
    第6页：蝙蝠侠追踪到哥谭港口，发现可疑货船...
    第7页：船上遭遇猫女，短暂交手后达成临时联盟..."
  user: [第8页图片]
```

**上下文构建规则**：

1. 只取前 5 页的 `plot`，不是全文翻译——减少 token 消耗
2. 如果某个 chunk 是第一块（如第 1-5 页），上下文为空——不编造
3. context 总长度控制在 800 字以内，超出则截断最早页
4. 如果某页缓存命中但 `plot` 为空（旧版缓存），跳过该页不报错

**设计决策**：不用全文翻译做上下文，原因：
- 全文翻译噪音大（包括拟声词、格式标记等）
- `plot` 是 AI 主动总结的，信息密度最高
- token 消耗可控（每页 summary 约 60-100 字）

### 3.4 耗时提示

**规则**：

```
if total_pages > 15:
    estimated_secs = total_pages * avg_time_per_page / concurrency
    对话框显示: "将翻译 XX 页，按当前并发数预计约 X-XX 分钟，请耐心等待"
```

`avg_time_per_page` 取保守估计 8 秒（含 API 往返 + 解析）。

## 6. 风险与限制

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| SQLite 迁移丢失数据 | 低 | 翻译缓存丢失，用户需重新翻译 | 迁移前自动备份 JSON → `.bak`；迁移后在内存中做 JSON vs SQLite 条数比对 |
| 上下文导致 prompt 过长超 token 限制 | 低 | API 调用失败 | context 硬限制 800 字；超过截断 |
| 上下文传递"错误剧情"（前页翻译本身有误） | 中 | 错误级联传播 | v3.1 先做基础传递；v3.2 的回溯修正会解决此问题 |
| 批量翻译中途 API 故障 | 中 | 部分页失败 | 失败页标记，不阻断后续；结束后列出失败清单，支持手动重试 |
| 并发写入 SQLite 冲突 | 低 | 数据丢失 | SQLite WAL 模式 + 写入重试（最多 3 次） |
| 存储层切换导致回归 bug | 中 | 单页翻译出现异常 | `database.py` 提供与旧 `cache.py` 完全兼容的接口；先跑通单页再开发批量 |

## 7. 不做的事

- **图片嵌字/叠加**：之前尝试过并失败，脱离项目「简单出翻译结果」的初衷
- **成本估算显示的金额**：仅显示页数 + 预估耗时，不计算具体金额
- **本地模型支持（Ollama）**：不考虑
- **翻译结果导出（TXT/MD）**：v3.1 不纳入，后续再看
- **术语表的手工编辑 UI**：v3.2 的自动术语表也是被动积累，不需要用户手动维护
