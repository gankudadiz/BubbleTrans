**日期**：2026-07-15  
**版本**：v3.1.2（计划中）  
**状态**：待实施  
**前置依赖**：v3.1（已实施）

## 1. 问题描述

用户清理 C 盘时，发现项目运行中或重启后点击翻译/翻页出现"加载失败"错误。

### 用户报告原文

> 重大bug，我在清理c盘才发现的，我点击翻译，看图片所示，说我加载失败，无法解码，因为图片缓存被删，我这个项目打开文件的图片缓存会被缓存到c盘？？？？这是绝对绝对不允许的

### 表象

1. 文件浏览器中仍能看到文件名（列表项来自首次加载时的内存缓存，未实时刷新磁盘）
2. 点击文件 → `QPixmap(image_path)` → `isNull()` → `workers.py:54` emit `"无法解码: C:\Users\...\Temp\bubbletrans_arc_xxx\xxx.jpg"`
3. `window.py:801` 收到错误 → 画布显示 **"加载失败"** 占位图

---

## 2. 根因分析

### 2.1 直接原因：压缩包解压目标在系统 Temp 目录

**`src/utils/archive.py:44-48`**：

```python
def _get_archive_temp_dir(archive_path: str) -> str:
    archive_hash = hashlib.md5(archive_path.encode('utf-8')).hexdigest()[:12]
    return os.path.join(tempfile.gettempdir(), f"{TEMP_DIR_PREFIX}{archive_hash}")
    #                                   ^^^^^^^^^^^^^^^^^^
    #                                   Windows → C:\Users\MLTZ\AppData\Local\Temp
```

`tempfile.gettempdir()` 在 Windows 上解析为 `C:\Users\<用户名>\AppData\Local\Temp`。用户清理 C 盘（磁盘清理 / `%TEMP%` 手动清理）会把 `bubbletrans_arc_<hash>` 目录连同解压出的所有图片一并删除。

### 2.2 次级原因：框选翻译临时文件也在系统 Temp

**`src/engine/translation_controller.py:51-53`**：

```python
temp_dir = tempfile.gettempdir()  # C:\Users\MLTZ\AppData\Local\Temp
unique_name = f"bubbletrans_{uuid.uuid4().hex[:8]}.png"
temp_path = os.path.join(temp_dir, unique_name)
```

框选翻译的临时截图同样写入 C 盘 Temp。虽然正常完成路径会清理（`_clean_temp_files()`），但：

1. 清理 C 盘瞬间若恰好有框选翻译在进行中，会出错
2. **当前 `_on_error` 路径未调用 `_clean_temp_files()`**（`translation_controller.py:104-110`），失败残留会留在 Temp；迁到项目 `cache/` 后更显眼，本次一并修

### 2.3 已知债务：缓存键含绝对路径（本次不修）

**`src/utils/cache.py:73-88`** — 缓存键 = `MD5(绝对路径 + int(mtime))`。`translation_cache.db` 中存储的 `image_path` 也是完整绝对路径。

| 场景 | 影响 |
|------|------|
| C 盘清理后 | image_path 指向的文件不存在，旧 Temp 路径缓存作废 |
| 换用户名 / 换电脑 / 程序搬家 | 绝对路径变化，缓存作废 |
| 压缩包路径变化 | `arc_<hash>` 的 hash 来自 archive 绝对路径，路径变则目录变，缓存 miss |

**本条仅作背景说明。** v3.1.2 **不修改** `_make_key()`、不做相对路径规范化、不写迁移工具。详见 §6。

### 2.4 故障触发链路（以压缩包为例）

```
用户打开 .cbz 压缩包
  → archive.py 解压到 C:\Users\...\Temp\bubbletrans_arc_<hash>\
  → file_browser.py 将 _current_folder 设为该 Temp 路径
  → canvas.py QPixmap(image_path) 从 Temp 加载图片
  → 翻译结果缓存到 SQLite，image_path = Temp 绝对路径

[用户清理 C 盘]
  → C:\Users\...\Temp\bubbletrans_arc_* 被删除

[用户再次操作]
  → file_list 仍显示文件名（QListWidget 有旧数据）
  → 点击文件 → os.path.join(_current_folder, filename) → 路径不存在
  → QPixmap 加载空文件 → "无法解码" → "加载失败"
```

## 3. 修复设计

### 3.1 核心原则

**项目产生的所有临时/缓存文件，一律存放在「应用数据根」下的 `cache/`，绝不触碰系统 Temp。**

应用数据根（与 `translation_cache.db` / `config.json` 同级）：

| 运行方式 | 根目录 |
|----------|--------|
| 开发（`python main.py`） | 项目仓库根（`src/utils/*.py` 上溯两级） |
| 打包（PyInstaller，`sys.frozen`） | `os.path.dirname(sys.executable)`（exe 旁） |

**禁止**仅用 `__file__` 推导根路径：打包后 `__file__` 落在临时解压目录（`_MEI*`），会再次把缓存写回 C 盘。

### 3.2 方案：本地 cache 目录

```
<应用数据根>/
├── cache/                        # 新增：本地缓存目录
│   ├── arc_<hash>/               # 压缩包解压图片（跨会话保留、可复用）
│   │   ├── page_001.jpg
│   │   └── page_002.jpg
│   └── bubbletrans_<uuid>.png    # 框选翻译临时截图（用完删除）
├── translation_cache.db
├── config.json
├── ...
```

`.gitignore` 中添加 `cache/`（开发态仓库根下的 cache；已有 `.cache/` / `tmp/` / `temp/` 勿误改）。

### 3.3 具体改动

#### 改动 1：统一「应用数据根 / cache 根」路径

新增可复用的根路径函数（优先放在已有工具模块，或抽到 `src/utils/paths.py`；**避免** `engine` 仅为路径去耦合 `archive` 业务）：

```python
def get_app_root() -> str:
    """与 config.json / translation_cache.db 同级的应用数据根"""
    if getattr(sys, 'frozen', False):
        return os.path.dirname(os.path.abspath(sys.executable))
    # 开发态：以调用方模块位置上溯，或固定从本模块推导到仓库根
    ...

def get_project_cache_dir() -> str:
    """<应用数据根>/cache/，不存在则创建"""
    cache_dir = os.path.join(get_app_root(), "cache")
    os.makedirs(cache_dir, exist_ok=True)
    return cache_dir
```

对齐现有模式：`src/utils/cache.py:50-55` 已对 `sys.frozen` 做了正确处理，本次必须复用同一规则。

#### 改动 2：`src/utils/archive.py` — 解压目标改为本地 cache + 可复用

| 位置 | 当前 | 目标 |
|------|------|------|
| `_get_archive_temp_dir()` | `os.path.join(tempfile.gettempdir(), ...)` | `os.path.join(get_project_cache_dir(), "arc_" + hash)` |
| 函数名 | `_get_archive_temp_dir()` | `_get_archive_cache_dir()` |
| 每次打开 | 若目录存在则 `rmtree` 再全量解压 | **优先复用**已有目录（见 §3.3.1） |
| `import tempfile` | 使用中 | 移除（若无其他用途） |
| `TEMP_DIR_PREFIX` | `bubbletrans_arc_` | 改为 `ARC_CACHE_PREFIX = "arc_"` 或内联 |

相关注释 / 函数名语义从「临时目录」改为「缓存目录」：`extract_to_temp` 可保留函数名以降低调用面改动，但 docstring 必须写明落点为项目 cache、可跨会话复用。

##### 3.3.1 解压复用策略（P0，必须实现）

**决策：二次打开同一压缩包时复用 `cache/arc_<hash>/`，禁止无条件 rmtree。**

当前代码（`archive.py:171-175` / `229-232`）在目录存在时总是清空重解压，若只改落盘路径、不改这段逻辑：

- 「shutdown 保留 cache 以便快速打开」完全无效
- 每次打开仍全量解压，磁盘 IO 与时间无收益
- CBR 重解压后 mtime 变化会导致翻译缓存 key 漂移（ZIP 有 `_restore_zip_mtime`，CBR 没有）

**打开压缩包时的逻辑：**

```
cache_dir = _get_archive_cache_dir(archive_path)
if cache_dir 存在且其中含至少 1 个图片文件:
    直接返回 (cache_dir, 排序后的图片文件名列表)   # 跳过解压
else:
    若 cache_dir 存在但不完整/为空 → 清理后重建
    执行解压
    返回 (cache_dir, image_files)
```

强制重解压场景（本次可不做 UI，仅代码预留或内联）：目录损坏、无图片、解压失败后的重试。

#### 改动 3：`src/engine/translation_controller.py` — 框选临时文件改为本地

| 位置 | 当前 | 目标 |
|------|------|------|
| `translate_region()` | `temp_dir = tempfile.gettempdir()` | `temp_dir = get_project_cache_dir()` |
| import | `import tempfile` | 移除，改用路径工具 |
| 清理 | 仅 finished / cancel / shutdown | **error 路径也调用 `_clean_temp_files()`** |

注意：框选临时文件翻译完成后应删除，不堆积。`_on_error` 当前未清理，本次必须补上。

#### 改动 4：`src/ui/file_browser.py` — 清理策略

| 时机 | 行为 |
|------|------|
| `shutdown()` | **不删除** `cache/arc_*`，保留以便下次复用 |
| `_load_source()` 切换来源 | 仅清理**上一个** `_archive_temp_dir`（若存在且路径不同） |
| 打开同一压缩包 | 不先删；交给 `archive` 复用逻辑 |

命名可保留 `_cleanup_archive_temp` / `_archive_temp_dir`，或逐步改为 cache 语义；行为以本表为准。

#### 改动 5（可选增强，v3.1.2 不做）：缓存键相对路径标识

见 §2.3 / §6。若未来做：对 `cache/arc_<hash>/` 下文件映射为 `archive_path::filename` 等稳定键。

### 3.4 影响范围分析

#### 3.4.1 兼容性

| 场景 | 影响 |
|------|------|
| 旧缓存（Temp 路径） | 路径变化后旧缓存全部失效，用户需对曾缓存页重新翻译。**可接受**，迁移必然代价。 |
| 普通文件夹模式 | 不受影响，`_current_folder` 本身不在系统 Temp。 |
| 跨设备 / 搬家 / 换用户名 | **本次不改善**。缓存键与 `arc_<hash>` 仍依赖绝对路径；仅保证不再写入系统 Temp。 |
| 打包发行版 | 必须把 cache 写到 **exe 旁**，不能写到 `_MEI*` 或 `%TEMP%`。 |

#### 3.4.2 清理策略变化

| 项目 | 当前 | 修复后 |
|------|------|--------|
| 压缩包目录清理 | shutdown + 切换来源时删除；且每次打开先 rmtree | shutdown **不删**；切换来源时只删上一份；**同一压缩包复用目录** |
| 手动清理方式 | 清理 C 盘 | 清理应用数据根下 `cache/` |
| 磁盘占用 | C 盘 | 与应用同盘（项目盘 / 便携目录） |
| 框选失败残留 | 可能留在系统 Temp | 应在 error/cancel/shutdown 删除；可选启动时扫掉 `cache/bubbletrans_*.png` |

**设计决策（已拍板）：**

1. 用户关闭程序时，保留 `cache/arc_<hash>/`
2. 用户再次打开**同一**压缩包时，**复用**该目录，不重解压
3. 用户切换到**不同**压缩包/文件夹时，清理**上一个** `arc_*` 目录
4. 不在本次做「多压缩包 LRU 长期保留多个 arc 目录」；当前会话模型仍是「同时只挂一份解压缓存」

#### 3.4.3 `_build_file_list()` 磁盘依赖

当前 `file_browser.py:262` 的 `os.listdir(self._current_folder)` 直接读磁盘。修复后 `_current_folder` 指向 `cache/arc_<hash>/`，不再依赖 C 盘 Temp。列表构建逻辑本身无需改。

## 4. 验收标准

| # | 验收项 | 通过条件 |
|---|--------|----------|
| A1 | 压缩包解压到本地 cache | 打开 `.cbz` / `.zip` / `.cbr`，图片在 `<应用数据根>/cache/arc_<hash>/`，**不在** `%TEMP%` |
| A2 | 翻译缓存跨会话有效（ZIP/CBZ） | 翻译 ≥1 页 → 关闭 → 重开同一压缩包 → 日志出现 `缓存命中: ...`；且**未再次全量解压**（复用目录） |
| A2b | CBR 跨会话 | 同 A2；因复用目录，mtime 不因重解压漂移，缓存应可命中 |
| A3 | 程序关闭后解压文件保留 | 关闭后 `cache/arc_<hash>/` 仍在且含图片 |
| A4 | 切换来源时旧缓存正确清理 | 打开 A.cbz → 再打开 B.cbz → A 对应 `cache/arc_<hash_a>/` 被删除；B 的目录存在 |
| A5 | 框选临时文件在本地 | 框选翻译时文件落在 `cache/bubbletrans_*.png`；**成功与失败**后均应删除 |
| A6 | 普通文件夹模式不受影响 | 打开普通图片文件夹，翻页和翻译行为不变 |
| A7 | C 盘清理不影响程序 | 清理 `%TEMP%` 后，已打开漫画仍可翻页/翻译；重启后再次打开同一压缩包仍可用（走 cache 复用） |
| A8 | 旧 Temp 缓存平滑降级 | 旧 `translation_cache.db` 中 Temp 路径不会误命中；新路径可正常写入 |
| A9 | 打包/便携路径 | 以 frozen 或实际 dist 启动时，cache 建在 **exe 同级** `cache/`，不在 `_MEI*` / 系统 Temp |

## 5. 风险与限制

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 旧缓存全部失效 | 100% | 用户需重新翻译已缓存的压缩包页 | 迁移代价无法避免；详见 §5.1 |
| 磁盘占用感知变化 | 中 | 应用目录变大（单压缩包解压可达数百 MB） | 切换来源时删上一份；用户可手动删 `cache/` |
| 崩溃后残留 `arc_*` / 框选 PNG | 低 | 占盘 | 切换时清理上一份；error 路径删框选临时文件；可选启动清 `bubbletrans_*.png` |
| 项目/安装目录只读 | 低 | 无法创建 `cache/` | `makedirs` 失败向上抛出，UI 层应有可读错误（沿用现有解压失败弹窗路径） |
| 并发双开同包 | 低 | 同目录写入冲突 | 现状为单实例使用假设，本次不引入文件锁 |

### 5.1 旧缓存失效说明

修复前，`translation_cache.db` 中已有缓存的 `image_path` 形如：

```
C:\Users\MLTZ\AppData\Local\Temp\bubbletrans_arc_rpe8wt2y\Batman 001.jpg
```

修复后，同一张图片的路径变为：

```
D:\漫画翻译器\BubbleTrans\cache\arc_rpe8wt2y\Batman 001.jpg
```

（打包态则为 `<exe目录>\cache\arc_...`）

缓存键 `MD5(path + int(mtime))` 不同，旧缓存不命中。**这是迁移必然代价。**

## 6. 不做的事

- **不改动缓存 key 生成算法**（`_make_key` 的 mtime 取整逻辑不变）：相对路径标识是后续增强，本次只解决「不写系统 Temp + 可复用解压目录」
- **不增加缓存迁移工具**：旧缓存条数少，手动重翻成本极低
- **不修改 SQLite 表结构**
- **不做多压缩包长期 LRU 缓存池**：仍只保留「当前来源」对应的一份 `arc_*`（shutdown 后磁盘上可残留最近一次，供下次同包秒开）
- **不在本次做「强制重新解压」UI 按钮**

## 9. 评审结论摘要（文案修订依据）

实施前必须满足：

1. **P0**：复用 `cache/arc_<hash>/`，禁止「只搬家、仍每次 rmtree 重解压」
2. **P0**：应用根路径支持 `sys.frozen`，与 `translation_cache.db` 同级
3. **P1**：不夸大跨设备收益；框选 error 路径清理；验收含打包路径与二次打开复用
