# BubbleTrans 预翻译功能设计

**创建日期**：2026-07-05
**当前状态**：已实施（commit 8dc65c8）

---

## 1. 背景

### 使用痛点

当前翻译流程是单任务串行阻塞的：

1. 翻到一页漫画 → 按 F5 翻译当前页 → 等 30~40 秒
2. 在这 30~40 秒等待中，用户翻到下一页看画 → 视觉上已被"剧透"
3. 翻回来等翻译结果 → 阅读体验被割裂

核心矛盾：**API 响应太慢（30~40s），翻页看画太容易剧透，两者不同步。**

### 解决思路

在当前页翻译的同时，**在后台并发翻译后续 N 页，结果写入缓存**。等用户翻到那些页时，缓存直接命中、秒出结果。

---

## 2. 方案

### 2.1 核心架构：PrefetchManager

新增组件 `PrefetchManager` 管理翻译队列和并发控制：

```
┌─────────────────────────────────────────────────────────┐
│                   PrefetchManager                        │
│                                                          │
│  queue: [第7页, 第8页, 第9页]           ← 待翻译队列      │
│  active: {                               ← 正在翻译（最多2个）│
│    第6页 → TranslationWorker_1,                          │
│    第5页 → TranslationWorker_2,                          │
│  }                                                       │
│  completed: {第4页, 第3页}               ← 已完成          │
│                                                          │
│  并发上限: 2（可配置）                                    │
└──────────────────────────────────────────────────────────┘
```

工作流：

1. `enqueue(image_paths)` → 去重后加入队列（跳过缓存命中 + 已在队列/active 中的）
2. 队列调度 → 取最多 2 个启动 `TranslationWorker`
3. 每个 worker 完成 → 结果写入翻译缓存 → 从队列取下一个启动
4. 全部完成 → 发射 `all_completed` 信号

线程安全说明：

`TranslationWorker` 继承 `QThread`，其 `finished` 信号通过 Qt 事件循环自动排队投递到主线程。PrefetchManager 的完成回调在**主线程**执行，对 `queue` / `active` / `completed` 的读写不存在竞态，无需额外加锁。

依赖注入：

- `TranslationCache`：构造时注入 `cache` 实例（由 MainWindow 持有），用于去重判断和结果写入
- 总页数：不自行维护，由 `enqueue()` / `resync()` 调用方传入已过滤好的路径列表；MainWindow 负责计算目标范围

### 2.2 结果分流

| 场景 | 翻译结果去哪？ |
|------|---------------|
| 按 F5 翻译当前页（主翻译） | → UI 面板显示 + 写缓存（**现有逻辑不变**） |
| 预翻译的页面（非当前页） | → 只写缓存，不更新 UI |
| 预翻译完成后，用户恰好翻到该页 | → 缓存命中，**秒出**（`load_image` 现有逻辑已支持） |

### 2.3 触发方式

| 方式 | 操作 | 行为 |
|------|------|------|
| **手动预翻译** | 右侧面板「预翻译 ▼」下拉按钮 → 选择下 N 页 | 立即入队 N 页 |
| **快捷键** | `Ctrl+Shift+F5` | 等同于「预翻译下 3 页」 |
| **自动预翻译** | Settings 开启「翻页时自动预翻译后 N 页」 | 每次翻页后自动入队 |

### 2.4 UI 设计

- **右侧面板**：「翻译当前页」按钮右侧新增下拉按钮 `预翻译 ▼`，选项：下 1 页 / 下 3 页 / 下 5 页
- **状态栏**：预翻译进行中显示 `📦 预翻译: 2/5 页已完成，1 页翻译中…`
- **Settings**：新增复选框「翻页时自动预翻译」+ 下拉框「预翻译页数（3/5 页）」+ 下拉框「并发数（1/2/3）」
- **预翻译完成后**：状态栏提示 `📦 预翻译完成: 5 页已缓存`，3 秒后消失

### 2.5 配置字段（config.json）

```json
{
  "prefetch_enabled": true,
  "prefetch_count": 3,
  "prefetch_concurrent": 2
}
```

## 3. 边界处理

### 3.1 尾部边界

| 场景 | 处理 |
|------|------|
| 当前第 98/100 页，预翻译下 3 页 | 只入队第 99、100 页（共 2 页），不越界 |
| 当前是最后一页 | 预翻译按钮置灰 / 提示"已是最后一页" |
| 总共只有 3 页，选预翻译下 5 页 | 只入队实际存在的后 2 页 |
| 总共只有 1 页 | 预翻译按钮不显示 |

### 3.2 缓存去重

| 场景 | 处理 |
|------|------|
| 页面已在翻译缓存中 | 跳过，不入队（避免浪费 API 调用） |
| 页面已在队列中或正在翻译 | 跳过，不重复入队 |

### 3.3 提升（主翻译抢占）

用户按 F5 翻译当前页，而该页恰好在预翻译队列中：

- 如果该页在队列中（未开始） → 从队列移除，交由当前页翻译处理
- 如果该页正在翻译中 → **不终止**，预翻译完成后正常写入 UI 面板

### 3.4 快速连续翻页（自动预翻译模式）

翻页时的队列重新同步逻辑：

- 翻到第 5 页 → 入队第 6~8 页
- 立即翻到第 6 页 → 第 6 页从队列提升（如未开始），新范围变为第 7~10 页
- 立即翻到第 8 页 → 移除队列中第 7 页（已翻过），追加第 11~13 页

策略：**每次翻页重新计算目标范围 [current+1, current+N]，队列中不在范围内的移除，新增的追加，已存在的保留。**

### 3.5 中断与清理

| 场景 | 处理 |
|------|------|
| 预翻译进行中，切换文件夹/压缩包 | 清空队列 + 终止所有进行中的 `TranslationWorker` |
| 预翻译进行中，关闭程序 | `closeEvent` 中统一终止并清理 |
| 预翻译过程中某页 API 报错 | 该页静默失败，继续处理队列中后续页面（不弹错误框） |

## 4. 涉及改动的文件

| 文件 | 改动 | 类型 |
|------|------|------|
| `src/ui/prefetch.py` | **新建** — `PrefetchManager` 类（队列 + 并发控制 + 信号） | 新增 |
| `src/ui/window.py` | 新增预翻译 UI 按钮/菜单 + 连接 PrefetchManager + 翻页自动预翻译逻辑 + `_set_translating` 适配多 worker | 修改 |
| `src/ui/settings.py` | 新增「自动预翻译」「预翻译页数」「并发数」配置项 | 修改 |
| `src/utils/config.py` | 新增 `prefetch_enabled` / `prefetch_count` / `prefetch_concurrent` 字段 + 默认值 | 修改 |

**不受影响**：`llm.py`、`cache.py`、`canvas.py`、现有翻译缓存逻辑。
