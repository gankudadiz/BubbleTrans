# BubbleTrans v3.0 架构重构 PDR

**日期**：2026-07-07
**版本**：v3.0（已完成）
**状态**：已完成
**前置**：无
**后续**：v3.1（多图批量翻译与上下文连贯，依赖本版本重构完成）

---

## 1. 升级动机

v2.4 功能完整可用，但代码架构存在结构性债务。继续在现有结构上堆功能（批量翻译、上下文传递、SQLite 迁移），将导致：

- `window.py` 从 1579 行轻松突破 2000 行
- 新增功能没有干净的代码承载点
- `prefetch → window` 循环依赖阻碍独立测试和维护
- SQLite 迁移需要在全文件散落的 6+ 处缓存调用中逐一修改

**v3.0 不做任何用户可见的功能变更。** 这是一次纯架构手术——拆解 God Object，建立清晰分层，为 v3.1 的功能升级铺路。

## 3. 重构方案

### 3.1 优先级排序（仅 P0 纳入 v3.0）

v3.0 聚焦 **P0 四步**——投入产出比最高的重构，解决核心结构性债务。P1/P2 可在后续版本中穿插进行。

> **路径约定**：下文 R1-R4 中的文件路径均相对于项目根目录下的 `src/` 目录（如 `engine/workers.py` 实际位于 `src/engine/workers.py`）。执行命令前请先 `cd src`。

#### R1：提取 Worker 类到 engine 层（解决循环依赖）

**目标**：`TranslationWorker` 和 `ImageLoadWorker` 从 `window.py` 移入新文件 `engine/workers.py`。

**改动文件**：
| 文件 | 操作 | 说明 |
|------|------|------|
| `engine/workers.py` | **新建** | `TranslationWorker(QThread)` + `ImageLoadWorker(QThread)` |
| `ui/window.py` | 删除两个 Worker 类定义，改为 `from engine.workers import ...` | ~80 行移除 |
| `ui/prefetch.py` | 改为 `from engine.workers import TranslationWorker` | 消除循环依赖 |

#### R2：提取图片缓存管理器

**目标**：`pixmap_cache`（LRU OrderedDict）+ 异步加载 + 预加载逻辑从 `window.py` 移入新文件 `ui/image_cache.py`。

**抽取内容**（约 150 行）：
- `pixmap_cache` OrderedDict 及相关常量
- `pending_loads` 集合
- `_add_to_cache()`
- `_prefetch_adjacent()`（预加载前后各 2 页，路径列表计算仍留在 window.py）
- `_on_prefetch_loaded()` 回调
- `_clear_image_cache()`

**新文件接口**：
```python
class ImageCacheManager(QObject):
    def get(self, path) -> QPixmap | None   # 查缓存并更新 LRU 位置
    def load_async(self, path)               # 主动异步解码
    def prefetch(self, paths)                # 后台预加载
    def clear(self)                          # 清空缓存 + 终止 worker
    # 信号: image_loaded(path, pixmap)
    # 信号: image_error(path, error_msg)
    # 信号: prefetch_completed(path, pixmap)
```

#### R3：提取翻译编排控制器

**目标**：翻译任务的启动、状态管理、缓存写回、Worker 生命周期集中到 `engine/translation_controller.py`。

**抽取内容**（约 200 行）：
- `translate_current_page()` 入口
- `handle_region_selected()` 框选翻译
- `on_translation_finished()` 结果处理 + 缓存写入
- `on_translation_error()` 错误处理
- `_set_translating()` 状态切换
- `_cleanup_worker()` Worker 安全清理
- 临时文件管理

**新文件接口**：
```python
class TranslationController(QObject):
    def translate_page(self, image_path: str)
    def translate_region(self, image_path: str, region_pixmap: QPixmap)
    def is_translating(self) -> bool
    def clear_page_cache(self, image_path: str) -> bool
    def clear_all_cache(self) -> int
    def cancel()
    def shutdown()
    # 信号: stage_changed(str)
    # 信号: translating_changed(bool)
    # 信号: translation_finished(origin, translated, summary)
    # 信号: translation_error(error_msg)
    # 信号: translation_started()
```

#### R4：canvas 封装修复 + 文件浏览器提取

**目标**：
1. 消除 `window.py` 中 3 处 `hasattr(self.canvas, '_current_image_path')`；
2. 将 `window.py` 中的文件列表、搜索过滤、最近菜单、压缩包打开等逻辑提取到 `ui/file_browser.py`。

**改动**：
- `canvas.py` 中增加公开属性：
```python
@property
def current_image_path(self):
    return getattr(self, '_current_image_path', None)
```
- 新建 `ui/file_browser.py`，包含 `FileBrowser(QWidget)` 类，通过信号与 `window.py` 通信。
- `window.py` 中保留 UI 布局、翻译面板、工具栏（Shortcuts/Settings），移除文件管理相关方法。

### 3.2 重构后架构

```
src/
├── main.py
├── engine/
│   ├── llm.py                        # 纯翻译逻辑（~400 行，移除配置管理和连接测试）
│   ├── workers.py                    # TranslationWorker + ImageLoadWorker（新建）
│   └── translation_controller.py     # 翻译编排 + 缓存写回 + 状态管理（新建）
├── ui/
│   ├── window.py                     # 纯布局 + 信号连接（目标 ~800 行）
│   ├── canvas.py                     # 图片显示 + 框选（加 @property 暴露路径）
│   ├── settings.py                   # 设置 UI
│   ├── image_cache.py                # QPixmap LRU 缓存 + 异步加载（新建）
│   ├── file_browser.py               # 文件列表 + 搜索 + 最近菜单（新建）
│   └── prefetch.py                   # 预翻译（修复 import 路径）
└── utils/
    ├── cache.py                      # 翻译缓存（加线程锁 R9）
    ├── config.py                     # 配置读写（加内存缓存 R10）
    └── archive.py                    # 压缩包（不变）
```

### 3.3 顺手修复的低成本项目

| # | 项目 | 说明 |
|---|------|------|
| R9 | `cache.py` 加 `threading.RLock` | 保护并发读写，一行锁的事 |
| R10 | `config.py` 加内存缓存 | 模块级 `_cached` 变量，避免频繁读盘 |
| R11 | 清理 `canvas.py` 死代码 | `_setup_nav_overlay` 从未被调用，整块删除 |

### 3.4 重构执行顺序

```
R1（提取 Worker）     ← 最先做，解除循环依赖
  ↓
R2（提取图片缓存）     ← 在 R3 之前执行
  ↓
R3（提取翻译控制器）   ← 在 R4 之前执行
  ↓
R4（canvas 封装修复 + 文件浏览器提取） ← 收尾
  ↓
R9/R10/R11 顺手修
```

**说明**：R2、R3、R4 都大量修改 `window.py` 的 `__init__`、`closeEvent`、回调函数等重叠区域，不建议并行执行，必须按顺序逐步合并，避免冲突。R9/R10/R11 与重构逻辑耦合度低，可在 R4 完成后任意顺序执行。

## 5. 风险与限制

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 重构引入 import 循环 | 中 | 程序无法启动 | 每步重构后立即启动验证；import 路径统一采用 `from engine.xxx import` 单向 |
| 信号连接断裂 | 中 | 翻译完成后 UI 不刷新 | 每个信号连接点写注释标注用途；重构后逐功能验证 |
| 动画/骨架屏异常 | 低 | 视觉异常但不影响功能 | 这些逻辑留在 window.py 不动，只移业务逻辑 |
| LRU 缓存行为变化 | 低 | 翻页性能下降 | `ImageCacheManager` 保留完全相同的 LRU 策略和容量 |
| Worker 生命周期管理改变 | 中 | 线程泄漏或重复翻译 | `TranslationController` 封装 Worker 完整生命周期，单例管理 |

## 6. 不做的事

本版本聚焦 P0 重构。以下明确不纳入：

- **P1 重构**：右侧面板独立、配置档案管理独立、连接测试独立 → v3.x 后续穿插
- **P2/P3 重构**：消除全局单例、引入依赖注入 → 远期优化
- **任何用户可见的功能变更**：这不是功能版本
- **SQLite 迁移**：v3.1 做（在干净架构上迁移）
